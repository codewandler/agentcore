## Goal

Evolve agentsdk's current `agentdir -> app.App` resource loading path into a
generic discovery, normalization, inspection, and import architecture.

The goal is broad compatibility without unsafe execution:

- detect agent-related resources from multiple ecosystems
- normalize safe declarative resources into agentsdk primitives
- keep executable or remote integrations disabled until explicitly trusted
- preserve the current `.agents`/`.claude` and app manifest behavior
- support global user resource injection while allowing callers to disable it
- support loading the same resource formats from local directories and, as a
  near milestone, git repositories
- prefer clean architecture over backwards compatibility; this design may
  replace current prototype APIs instead of preserving awkward compatibility
  shims

## Current State

agentsdk already has a narrow version of the desired model:

```text
agentdir.ResolveDir   rough Detector + app manifest resolver
agentdir.LoadFS       Loader + Resolver for .agents/.claude/plugin roots
agentdir.Bundle       simplified contribution bundle
app.App               importer, registry, and runtime composition root
skill.Repository      per-agent resolved skill repository
command.Registry      slash command primitive registry
terminal/cli          built-in run command over one resource source
```

This works for miniagent and native SDK bundles, but it does not yet model:

- candidate resources before import
- source identity and namespace
- trust level
- disabled executable resources
- diagnostics
- cross-ecosystem adapters
- global versus local discovery policy
- inspection and explicit trust flow

## Terminology

- **Resource candidate**: a discovered path, embedded root, URL, or manifest
  entry that might contain resources.
- **Source reference**: stable metadata describing where a resource came from:
  ecosystem, root, path, scope, trust, and whether it is local/global/remote.
- **Contribution bundle**: normalized declarative contributions from one source:
  agents, skills, commands, tools, hooks, permissions, and diagnostics.
- **Adapter**: ecosystem-specific detector/loader/resolver implementation.
- **Importer**: takes normalized bundles and registers enabled contributions
  into an `app.App` registry.
- **Registry**: app-owned indexed view of agents, commands, skills, tools, and
  diagnostics after conflict resolution.
- **Inspection**: read-only listing of discovered, imported, disabled, and
  conflicting resources. Inspection must not enable execution by itself.

## Resource Names and Conflicts

Each contribution has:

- a **short name**, usually the resource-local name, such as `reviewer`
- a **qualified ID**, generated by agentsdk, for unambiguous selection
- source metadata, including ecosystem, scope, and path

Use a short but stable qualified ID format. The path/ref suffix is always part
of the qualified ID; do not add it only after a collision.

```text
<ecosystem>:<scope>:<name>#<stable-relative-path-or-ref>
```

Examples:

```text
agents:project:reviewer#agents/reviewer.md
agents:project:reviewer#.agents/agents/reviewer.md
claude:project:reviewer#.claude/agents/reviewer.md
agents:user:reviewer#.agents/agents/reviewer.md
agents:git:reviewer#github.com/codewandler/agentplugins//agents/reviewer.md
```

Short-name resolution remains available. First registered resource wins the
short name. Later resources with the same short name are still tracked by
qualified ID, ignored for short-name lookup, and reported as warnings.

Default registration order determines precedence. Built-in SDK discovery should
register:

1. explicit app manifest resources, in manifest order
2. project `.agents`
3. project `.claude`
4. project root plugin layout
5. global user `.agents`, only when enabled
6. global user `.claude`, only when enabled

Custom Go applications can provide their own ordered discoverer list. The first
discoverer that registers a short name wins. This lets a branded app define
its own precedence instead of hard-coding all policy inside agentsdk.

Example warning:

```text
warning: agent "reviewer" from claude:project:reviewer was ignored because
agents:project:reviewer already registered that short name
```

## Discovery Policy

`agentsdk run <dir>` should load resources from the pointed-to directory. By
default it should not include global user resources. Branded applications such
as miniagent may opt in by configuring the SDK loader. App manifests can also
control this behavior.

```text
<dir>/app.manifest.json or <dir>/agentsdk.app.json
<dir>/.agents/
<dir>/.claude/
<dir> as plugin root
~/.agents/   only when global user discovery is enabled
~/.claude/   only when global user discovery is enabled
```

Add a flag/config option to include discovery outside the specified path:

```text
agentsdk run <dir> --include-global
```

Possible SDK-level config names:

```go
DiscoveryPolicy{
    IncludeGlobalUserResources bool
    IncludeExternalEcosystems  bool
    AllowRemote                bool
    TrustStoreDir              string
}
```

Default recommendation:

- local specified directory: enabled
- local project `.agents`/`.claude`: enabled
- global user `.agents`/`.claude`: disabled for generic `agentsdk run`
- branded apps can enable global user resources by default
- app manifests can enable or disable global user resources explicitly
- broad external ecosystem discovery: disabled unless inspection mode or a
  manifest/app policy enables it
- remote discovery/import: disabled unless explicitly configured

Initial manifest shape:

```json
{
  "default_agent": "main",
  "discovery": {
    "include_global_user_resources": true,
    "include_external_ecosystems": false,
    "allow_remote": false,
    "trust_store_dir": ".agentsdk"
  },
  "sources": [
    ".agents",
    "file:///abs/path/to/resources",
    "git+https://github.com/codewandler/agentplugins.git",
    "git+ssh://git@github.com/codewandler/agentplugins.git"
  ]
}
```

Manifest sources should be URL-like strings:

- bare strings without a protocol are local paths relative to the manifest
- `file://...` is an explicit local filesystem source
- `git+https://...` is a git source over HTTPS
- `git+ssh://...` is a git source over SSH

This keeps manifests simple while still leaving room for query parameters or
fragments later, for example refs or subdirectories if needed.

Git source cache layout:

```text
<workspace>/.agentsdk/
  cache/
    git/
      <host>/
        <owner>/
          <repo>/
            refs/
              <sanitized-ref>/
                worktree/       materialized checkout used for loading
                meta.json       source URL, resolved commit, fetched time
```

Example:

```text
.agentsdk/cache/git/github.com/codewandler/agentplugins/refs/main/worktree/
```

The cache path is derived from normalized source URL plus ref. Materialized git
sources are still treated as declarative resource roots; repository code is not
executed just because it exists in the cache.

## Security Policy

Follow broad detection, narrow execution:

- Markdown/YAML/JSON/TOML declarative resources may be parsed and shown.
- Local commands discovered from `Makefile`, `Taskfile.yml`, `package.json`,
  `justfile`, etc. are not auto-registered as executable commands.
- Dev commands can be surfaced as diagnostics/suggestions with an easy path to
  add/approve them.
- Remote resources require explicit approval before import.
- Plugins are resource bundles, not directly executed code.
- Go-defined plugins/tools remain explicit application code and enter through
  `app.WithPlugin` or equivalent trusted APIs.
- Rule/trust persistence requires explicit configuration. A reasonable default
  store for CLI workflows is `<workspace>/.agentsdk/`. Applications should be
  able to choose another path or disable persistence, but the generic CLI uses
  `<workspace>/.agentsdk/` by default.

## Target Package Shape

### New `resource` Package

Own the normalized model:

```go
type TrustLevel string

const (
    TrustUntrusted TrustLevel = "untrusted"
    TrustDeclarative TrustLevel = "declarative"
    TrustTrusted TrustLevel = "trusted"
)

type Scope string

const (
    ScopeProject Scope = "project"
    ScopeUser    Scope = "user"
    ScopeRemote  Scope = "remote"
    ScopeEmbedded Scope = "embedded"
    ScopeGit     Scope = "git"
)

type SourceRef struct {
    ID        string
    Ecosystem string
    Scope     Scope
    Root      string
    Path      string
    Ref       string
    Trust     TrustLevel
}

type DiscoveryPolicy struct {
    IncludeGlobalUserResources bool
    IncludeExternalEcosystems  bool
    AllowRemote                bool
    TrustStoreDir              string
}

type Candidate struct {
    Source SourceRef
    Kind   string
}

type Diagnostic struct {
    Severity string
    Source   SourceRef
    Message  string
}

type ContributionBundle struct {
    ID          string
    Name        string
    Source      SourceRef
    Agents      []AgentContribution
    Skills      []SkillContribution
    Commands    []CommandContribution
    Tools       []ToolContribution
    Hooks       []HookContribution
    Permissions []Permission
    Diagnostics []Diagnostic
}
```

### New `resource/discovery` or `discovery` Package

Own orchestration interfaces:

```go
type Detector interface {
    Name() string
    Detect(context.Context, []Root) ([]resource.Candidate, error)
}

type Loader interface {
    Kind() string
    Load(context.Context, resource.Candidate) (*NativePackage, error)
}

type Resolver interface {
    Resolve(context.Context, *NativePackage) (*resource.ContributionBundle, error)
}
```

Discovery orchestration must accept custom ordered discoverers:

```go
type Discoverer interface {
    Name() string
    Discover(context.Context, Request) ([]resource.ContributionBundle, error)
}

type Request struct {
    Roots  []Root
    Policy resource.DiscoveryPolicy
}
```

Custom applications can pass discoverers in desired precedence order. The first
registered contribution for a short name wins.

### Adapter Packages

Keep adapters independent:

```text
agentdir/ or adapters/agentdir/     native .agents + Claude-compatible layout
adapters/git/                       clone/fetch git sources, then run format adapters
adapters/codex/                     AGENTS.md/config metadata detection, later
adapters/skills/                    **/SKILL.md discovery, later
adapters/mcp/                       mcp.json detection, no auto-connect, later
adapters/devcmd/                    Makefile/Taskfile/package scripts suggestions, later
adapters/openapi/                   OpenAPI tool candidates, disabled by default, later
```

Initial implementation should refactor the existing native/Claude path and then
add git loading for the same `.agents`/`.claude` resource formats. Do not add
A2A, OpenClaw, or broad ecosystem adapters until the normalized model, local
format adapter, and git source handling are solid.

## App Integration

`app.App` should import normalized bundles, not `agentdir.Bundle` directly.

Current:

```go
app.WithBundle(agentdir.Bundle)
```

Target:

```go
app.WithResourceBundle(resource.ContributionBundle)
app.WithResourceRegistry(*resource.Registry)
```

Migration should prefer the target API over compatibility. Temporary internal
bridges are acceptable only while moving the code within one iteration; do not
preserve awkward public compatibility shims.

- `agentdir` emits `resource.ContributionBundle`
- app imports that bundle
- public APIs should move toward `resource`

Registry responsibilities:

- store all contributions with source refs
- enforce trust policy
- resolve short names and qualified names
- collect diagnostics
- expose app-specific views:
  - user-callable commands
  - agent-callable commands
  - selected/default agent
  - disabled suggestions

Tools must be part of the contribution model even before SDK implements
automatic tool discovery. Tool contributions should initially be abstract
metadata:

```go
type ToolContribution struct {
    ID          string
    Name        string
    Description string
    Source      SourceRef
    Enabled     bool
    Metadata    map[string]any
}
```

Only trusted code-defined tools should become executable `tool.Tool` values.
Discovered tool candidates, such as OpenAPI operations or MCP tools, remain
disabled suggestions until an explicit adapter/trust path turns them into
runtime tools.

## Conflict Resolution

Implement deterministic conflict handling with diagnostics.

Rules to encode:

1. first registration wins for short-name lookup
2. later duplicates are ignored for short-name lookup and recorded as warnings
3. project resources should be registered before global resources
4. local resources should be registered before remote resources
5. `.agents` should be registered before `.claude` when no explicit order exists
6. explicit app manifest plugin order overrides built-in heuristics
7. all resources remain addressable through qualified IDs where possible
8. qualified IDs always include a stable path/ref suffix for maximum stability
